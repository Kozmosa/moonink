# MoonInk RFC

## 1. 摘要

MoonInk 是一个使用 MoonBit 构建的静态站点生成器，面向文档、博客与知识站点，第一版聚焦于个人知识发布场景。本文档定义 MoonInk 首个可实现版本的产品方向、范围边界、架构基线与核心工程决策。

MoonInk 将 MkDocs 作为参考模型，借鉴其可预测的配置方式、从内容到站点的生成思路，以及清晰的构建阶段划分；但 MoonInk 不追求完整复刻 MkDocs 的功能面。相反，第一版将强调一种更小、更易讲解、更可控的架构，使其既能够成为一个真实可用的工具，也能够成为 MoonBit 在开发工具工程能力上的示范项目。

## 2. 背景与动机

现有静态站点生成器已经较好地覆盖了文档与发布场景，但 MoonBit 仍缺少一个具有代表性的、端到端的、工具级项目，用来展示这门语言如何支撑配置解析、内容处理、渲染流水线以及命令行工作流等较为完整的工程系统。

因此，这个项目同时承担两个目标：

- 提供一个实用的、面向知识组织的静态站点生成器；
- 证明 MoonBit 可以支撑一个架构清晰、实现可维护的真实开发工具。

从应用视角来看，本项目要解决的问题不是“如何生成任意网站”，而是“如何以尽量低的概念负担，把结构化的个人知识内容变成一个可预测、可导航、可搜索的静态站点”。

## 3. 目标

- 构建一个基于 MoonBit 的静态站点生成器，服务个人知识作者。
- 支持双轨站点模型：文档树 + 文章流。
- 保持第一版以内容为先、行为可预测、规模足够克制，便于清晰实现。
- 复用 MkDocs 已被验证有效的理念，尤其是清晰的配置方式与分阶段构建流程。
- 产出可直接支撑项目申报的设计文档与工程文档。

## 4. 非目标

- 不追求与 `mkdocs.yml` 或整个 MkDocs 插件生态完全兼容。
- 第一版不提供完整的第三方插件系统。
- 第一版不提供主题市场或多主题包分发机制。
- 第一版不提供完整多语言支持。
- 第一版不提供增量构建与部署自动化。
- 第一版不提供知识图谱可视化或复杂的反向链接界面。

## 5. 目标用户与核心场景

### 5.1 目标用户

MoonInk 的首要目标用户是个人知识作者。他们希望将 Markdown 内容组织成结构化笔记、长篇文档与文章型条目，并发布为可长期维护的静态站点。

### 5.2 核心场景

第一版将针对如下知识库风格站点进行优化：

- 具有层级结构的文档或笔记树；
- 按时间或标签组织的文章流；
- 知识单元之间的内部链接；
- 面向已发布内容的本地静态搜索。

## 6. 产品定位

MoonInk 被定位为一个冷静、结构化、内容优先的生成器。其产品气质应与 `docs/Brand_Design.md` 中已经确定的品牌方向保持一致：克制、精确、安静、可持续。

在工程层面，这意味着：

- 优先使用简单配置，而不是灵活但难以推理的抽象；
- 优先采用显式的分阶段构建流水线，而不是不透明的“魔法”；
- 优先打磨强默认主题，而不是过早暴露尚未稳定的扩展面；
- 通过可预测行为建立质量，而不是通过堆叠功能数量建立表面能力。

## 7. 相对于 MkDocs 的参考策略

MoonInk 将 MkDocs 作为以下方面的参考：

- 项目级配置；
- 导航与内容组织理念；
- 构建命令工作流；
- 内容、主题与输出的分离方式。

MoonInk 在以下方面有意与 MkDocs 分化：

- 配置采用 TOML，而不是 YAML；
- 第一版范围更窄；
- 信息模型更明确地面向知识库场景；
- 架构文档会更强调可教学性与 MoonBit 实现导向。

## 8. 第一版范围

### 8.1 第一版包含内容

- TOML 配置文件；
- Markdown 内容输入；
- 支持核心字段 `title`、`author`、`id`、`tags`、`date` 的 frontmatter；
- 双轨站点结构：文档树 + 文章流；
- 混合导航策略：自动生成默认导航，并允许人工覆盖；
- 内部相对链接校验；
- WikiLink 语法支持；
- 单个官方内置主题；
- 静态搜索索引生成；
- CLI 命令：`moonink new`、`moonink build`、`moonink serve`；
- 单语言站点生成；
- 核心模块的单元测试。

### 8.2 明确延期内容

- 第三方插件；
- 多个官方主题；
- 正式的部署命令；
- 多语言站点生成；
- 增量构建缓存；
- 完整的反向链接或图谱视图；
- 内容组件或类似 MDX 的嵌入式 UI 语法。

## 9. 产品模型

MoonInk V1 将一个站点视为以下五层的组合：

1. 配置；
2. 内容源树；
3. 解析后的内容与元数据；
4. 站点模型；
5. 渲染后的输出制品。

这种分离很重要，因为它使实现过程可解释、可测试，并且能够自然映射到 MoonBit 的模块组织方式。

## 10. 核心设计决策

### 10.1 配置格式

MoonInk 第一版将使用 TOML 作为配置格式。TOML 在保持可读性的同时，能够提供比临时式 frontmatter 配置更清晰的结构，也避免项目在早期过度耦合到兼容 MkDocs 的 YAML 路线。

### 10.2 内容格式

第一版仅接受 Markdown。这可以控制解析、渲染与校验范围。以后如有需要，可再扩展到结构化数据驱动页面等能力。

### 10.3 站点结构

MoonInk 同时支持层级化知识树与文章流。这个双轨模型更符合真实的个人知识发布场景：有些内容属于稳定的概念结构，有些内容则属于随时间推进的写作流。

### 10.4 元数据模型

第一版元数据模型将有意保持克制。

- 必填：`title`
- 推荐：`id`、`date`
- 可选：`author`、`tags`

实现中可以预留 `summary`、`draft`、`order`、`aliases` 等未来字段，但这些都不作为第一版范围的必要条件。

### 10.5 链接能力

MoonInk V1 必须同时支持：

- 标准内部链接与路径校验；
- 基于页面标识规则解析的 `[[WikiLink]]` 语法。

这项决策使 MoonInk 在保持功能节制的同时，具备更明显的知识库友好性。

### 10.6 主题策略

V1 只提供一个官方主题。主题系统在架构上应设计为可替换，但暂不将其稳定为公共生态 API。

### 10.7 插件策略

V1 不开放第三方插件机制。但构建流水线与内部模块边界应当设计为未来可以引入生命周期钩子，而不需要大幅重构。

### 10.8 搜索

V1 在构建阶段生成本地静态搜索索引，并在生成站点中通过前端方式完成搜索。

### 10.9 国际化

V1 采用单语言策略。国际化支持应作为演进路径被文档化，而不应过早进入实现范围。

## 11. 架构概览

基础构建流水线如下：

1. 加载并校验配置；
2. 发现内容文件与静态资源；
3. 解析 frontmatter 与 Markdown；
4. 规范化页面标识与路由；
5. 解析链接与 WikiLinks；
6. 构造导航与站点模型；
7. 使用官方主题渲染页面；
8. 输出资源、HTML 与搜索索引。

之所以选择这样的架构，是因为它：

- 便于在申报材料中解释；
- 便于映射为 MoonBit 模块；
- 便于按阶段测试；
- 便于未来加入扩展点。

## 12. 命令行体验

第一版将暴露以下命令：

- `moonink new`：创建一个起始项目；
- `moonink build`：将站点构建到输出目录；
- `moonink serve`：启动本地开发服务器。

CLI 应呈现稳定、冷静、信息充分的体验，与 MoonInk 的品牌语气保持一致。

## 13. 质量基线

V1 的质量基线以单元测试为核心。优先测试区域包括：

- 配置解析与校验；
- frontmatter 解析；
- 路由生成；
- 导航生成；
- 内部链接校验；
- WikiLink 解析；
- 搜索索引生成。

## 14. 风险

### 14.1 生态风险

MoonBit 在内容工具链、解析库与模板能力上的生态成熟度还不如 Python 或 JavaScript，这可能增加基础模块的实现成本。

缓解策略：保持 V1 范围紧凑；优先使用内部抽象并尽量减少依赖；将模块设计为可渐进替换。

### 14.2 渲染复杂度风险

一个过于强大的主题或模板系统很容易快速膨胀出复杂性。

缓解策略：V1 只交付一个官方主题，并仅稳定内部接口。

### 14.3 知识库范围膨胀风险

知识库产品往往很快就会扩展到反向链接、图谱、引用、嵌入等大量笔记型功能。

缓解策略：V1 聚焦双轨组织、WikiLink 与搜索，将图谱类能力后置。

### 14.4 开发服务器风险

实时预览与文件监听往往会比预期更复杂。

缓解策略：先把 `serve` 定义为一个实用的本地预览命令，热重载质量后续逐步演进。

## 15. 备选方案分析

### 15.1 完整兼容 MkDocs

不采纳。原因是兼容性压力会在 MoonBit 原生设计尚未建立之前扭曲架构，并过早拉大项目规模。

### 15.2 以博客为先的产品路线

不采纳。因为本项目目标与结构化知识发布的契合度高于纯时间序写作。

### 15.3 以插件为先的架构路线

不采纳。因为在核心模型稳定之前优先追求扩展性，容易引入过早抽象。

## 17. RFC 方向验收标准

如果项目能够在进入实现前，对以下问题给出清晰答案，则 RFC 方向可视为成立：

- V1 包含什么，不包含什么；
- V1 首先服务谁；
- 架构阶段如何划分；
- CLI 暴露哪些命令；
- 内容模型如何定义；
- 哪些能力被有意后置。

## 18. 结论

MoonInk V1 应当是一个聚焦、内容优先、面向知识组织的 MoonBit 静态站点生成器。它应借鉴 MkDocs 的正确经验，但不继承其全部复杂度。只要它既能成为一个可用的发布工具，又能成为 MoonBit 在真实工程系统中的可信示范，这个项目就达成了目标。

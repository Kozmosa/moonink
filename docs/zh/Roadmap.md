# MoonInk Roadmap

## 1. 路线图定位

本路线图将 MoonInk RFC 转化为可执行的工程计划。它同时服务于项目实现与后续申报书撰写，因此每个阶段都会包含产品范围、技术产出、风险以及完成信号。

## 2. 总体开发原则

- 优先完成一条完整可用的纵向切片，而不是追求功能面铺开；
- 在开放扩展性之前，先稳定核心模型；
- 从一开始就保持明确的模块边界；
- 确保每个阶段都留下可复用的文档与可验证的制品。

## 3. Phase 0 - 基础设计与方案冻结

### 3.1 目标

将当前概念方案收敛为稳定的实现基线。

### 3.2 交付物

- RFC 文档定稿；
- Roadmap 文档定稿；
- `docs/technical/` 下完整技术文档集；
- 基于 MkDocs 与 MoonBit 文档的参考分析摘要；
- 第一版样例站点的信息架构方案。

### 3.3 工程重点

- 定义模块边界；
- 定义内容模型；
- 定义路由与导航规则；
- 定义配置模式与 CLI 合同。

### 3.4 退出条件

- 第一版范围问题没有悬而未决的关键项；
- 架构图与模块职责已经文档化；
- 实现顺序已经清晰到可以直接开始编码而无需重新设计。

## 4. Phase 1 - MVP 核心生成器

### 4.1 目标

交付一个具备最小完整能力集的可用静态站点生成器。

### 4.2 面向用户的范围

- `moonink new` 可初始化起始项目；
- `moonink build` 可生成静态站点；
- `moonink serve` 可进行本地预览；
- 支持带 frontmatter 的 Markdown 内容；
- 支持双轨内容结构；
- 可自动生成默认导航，并允许人工调整；
- 官方主题可以稳定渲染页面；
- 生成站点具备静态搜索能力。

### 4.3 核心工程任务

- 实现 TOML 配置加载与校验；
- 实现内容发现与文件分类；
- 实现 frontmatter 解析器；
- 实现 Markdown 解析流水线；
- 实现路由生成与页面标识规范化；
- 实现内部链接校验与 WikiLink 解析；
- 实现站点模型构建器；
- 实现渲染器与官方主题资源；
- 实现搜索索引输出；
- 实现 CLI 与本地开发服务器。

### 4.4 建议里程碑

- M1：配置 + 文件发现 + 起始项目模板；
- M2：Markdown + frontmatter 解析；
- M3：导航、路由生成与页面模型；
- M4：渲染器 + 官方主题；
- M5：搜索索引 + `serve` 命令 + 测试稳定化。

### 4.5 退出条件

- 样例站点能够成功构建；
- 生成站点同时包含文档页与文章页；
- 链接校验能够清晰报告错误；
- WikiLink 解析能在已知 fixture 上正确工作；
- 单元测试覆盖关键模块。

## 5. Phase 2 - 质量与生产力增强

### 5.1 目标

在不破坏 MVP 模型稳定性的前提下，提升性能、写作质量保障与操作完整性。

### 5.2 优先增强项

- 增量构建；
- 更强的 `check` 能力或构建前校验流水线；
- 更好的本地预览反馈；
- 更完善的搜索索引与排序；
- 部署集成目标。

### 5.3 次级增强项

- 更丰富的元数据字段；
- 更完善的文章列表与标签聚合；
- 更强的路由冲突诊断；
- 更多样例模板。

### 5.4 退出条件

- 大型 fixture 站点的重建速度快于完整清洁构建；
- 诊断信息对内容作者足够可操作；
- 部署工作流已经被文档化，并实现部分自动化。

## 6. Phase 3 - 国际化与发布扩展

### 6.1 目标

让 MoonInk 从单语言发布扩展到更广泛的使用场景。

### 6.2 计划范围

- 多语言内容目录或语言前缀；
- 本地化导航与元数据处理；
- 多语言路由与资源策略；
- 更正式的部署集成。

### 6.3 风险

- 路由复杂度会显著提升；
- 内容同步规则会更难处理；
- 主题层的一些假设可能需要被泛化。

### 6.4 退出条件

- 语言维度的构建结果具有确定性；
- 官方主题在本地化导航下能够正常工作；
- 多语言站点的项目结构已有明确文档说明。

## 7. Phase 4 - 扩展性与生态

### 7.1 目标

在核心架构被验证稳定之后，开放经过审慎设计的扩展点。

### 7.2 计划范围

- 将内部生命周期钩子正式化为插件模型；
- 主题覆盖或主题包机制；
- 部分官方扩展，如搜索增强、sitemap 或 RSS；
- 知识库增强能力，如反向链接或面向图谱的元数据。

### 7.3 退出条件

- 扩展 API 具备版本化规则；
- 至少有一个非核心扩展能够证明架构有效；
- 扩展失败模式可诊断且边界清晰。

## 8. 跨阶段技术重点

### 8.1 文档

每个阶段都应保持实现文档与实际代码行为一致。对于一个要展示工程清晰度的工具项目来说，文档漂移是重要的长期风险。

### 8.2 测试

测试从 Phase 1 的单元覆盖起步，并在后续阶段逐步扩展到基于 fixture 的回归检查与集成测试。

### 8.3 品牌一致性

官方主题、CLI 文案与样例项目都应体现 MoonInk 的品牌方向：克制、精确、安静而现代。

## 9. 路线图层面的整体风险

- MoonBit 的库支持情况可能限制解析器或模板实现方案；
- 搜索与开发服务器功能容易快速膨胀出复杂度；
- 知识库产品的预期会推动项目越过第一版边界；
- 多语言支持可能迫使我们重新审视路由与内容标识假设。

## 10. 成功指标

如果路线图最终能够导向以下结果，则可视为有效：

- 交付一个可工作的、基于 MoonBit 的静态站点生成器 MVP；
- 形成一套在项目申报中具有说服力的技术架构；
- 产出足以指导实现与未来协作者的文档；
- 给出从 MVP 走向可扩展生态的清晰升级路径。

## 11. 近期下一步动作

- 完成技术文档集；
- 将 Phase 1 进一步拆分为实现任务；
- 准备一个 MoonInk 样例项目布局；
- 在合适时机将 RFC 与 Roadmap 转写为申报书语言。
